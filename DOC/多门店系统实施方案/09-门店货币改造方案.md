# 多门店系统 - 门店货币改造方案

## 一、目标与范围

### 1.1 目标
在现有多门店体系基础上，为每个门店增加独立货币配置，并确保：
- mall 移动端按当前门店显示对应货币符号与金额格式。
- 管理后台（web-antd）涉及金额展示的页面按当前门店显示对应货币。
- 后端保持金额存储口径不变（建议继续以分为单位存储），仅增加“展示货币”维度。

### 1.2 范围
- 数据库：门店表 `mall_store` 增加货币字段。
- 后端：门店实体/VO/API 增加货币字段；增加统一货币解析能力。
- 前端（管理端）：门店管理新增货币配置；商城/会员/统计等金额显示页按门店货币展示。
- 移动端（lukdo-mall）：按门店上下文展示货币。

### 1.3 非目标（本期不做）
- 不改订单、支付、统计等历史金额数值本身。
- 不做跨币种自动汇率换算。
- 不做同一订单多币种混算。

## 二、数据库设计

## 2.1 表结构变更
基于 `sql/mysql/store/store.sql` 的 `mall_store`，新增：
- `currency_code`：ISO 4217 货币代码（如 `USD`、`KHR`、`CNY`）
- `currency_symbol`：货币符号（如 `$`、`៛`、`¥`）

建议 SQL（新增脚本命名建议：`17-store-currency.sql`）：

```sql
ALTER TABLE `mall_store`
  ADD COLUMN `currency_code` varchar(10) NOT NULL DEFAULT 'USD' COMMENT '货币代码（ISO 4217）' AFTER `logo`,
  ADD COLUMN `currency_symbol` varchar(10) NOT NULL DEFAULT '$' COMMENT '货币符号' AFTER `currency_code`;

ALTER TABLE `mall_store`
  ADD KEY `idx_currency_code` (`currency_code`);
```

## 2.2 存量数据回填
- 默认门店（id=1）按当前业务默认币种回填（建议 `USD/$`）。
- 其他门店可先统一回填 `USD/$`，后续由运营在门店管理页逐步修正。

示例：

```sql
UPDATE `mall_store`
SET `currency_code` = 'USD', `currency_symbol` = '$'
WHERE `currency_code` IS NULL OR `currency_code` = '';
```

## 2.3 回滚方案
新增回滚脚本：`17-store-currency-rollback.sql`

```sql
ALTER TABLE `mall_store`
  DROP INDEX `idx_currency_code`,
  DROP COLUMN `currency_symbol`,
  DROP COLUMN `currency_code`;
```

## 三、后端改造方案（lukdo-back）

## 3.1 领域模型与接口
涉及模块：`yudao-module-mall/yudao-module-store`

需要修改：
- `StoreDO` 增加 `currencyCode`、`currencySymbol`
- `StoreRespVO`、`StoreSaveReqVO`、`StoreSimpleRespVO` 增加上述字段
- `MallStoreApi.StoreSimple` 对应后端接口返回扩展（供前端/移动端读取）

## 3.2 参数校验与业务规则
- `currencyCode` 必填，统一大写，长度 <= 10。
- `currencySymbol` 必填，长度 <= 10。
- 可选：限制在白名单中（如 `USD/KHR/CNY/THB/VND`），避免脏数据。

## 3.3 门店上下文货币解析能力
新增统一服务（建议）：
- `StoreCurrencyService#getCurrentStoreCurrency()`
- 解析顺序：
1. `StoreContextHolder.getStoreId()`
2. 查询 `mall_store.currency_code / currency_symbol`
3. 查不到则返回系统默认（`USD/$`）

并新增轻量接口供移动端读取当前门店货币（可选）：
- `GET /app-api/mall/store/current-currency`
- 返回：`storeId/currencyCode/currencySymbol`

## 3.4 金额展示口径
- DB 与接口金额字段仍保留“分”或现有数值类型，不引入换算。
- 货币只用于前端展示层（符号、格式）。

## 四、管理端前端改造（lukdo-front / web-antd）

## 4.1 门店管理页改造
页面：`apps/web-antd/src/views/mall/store`

改造点：
- 表单新增字段：
  - `currencyCode`（Select，默认 `USD`）
  - `currencySymbol`（Input/自动回填，默认 `$`）
- 列表新增列：`货币代码`、`货币符号`（可放在“门店编码”后）

## 4.2 全局门店货币状态
建议新增全局状态（pinia/store）：
- `currentStoreCurrencyCode`
- `currentStoreCurrencySymbol`

来源：
- 复用当前左侧门店栏 `StorePanel` 的选中门店结果，直接写入全局。
- 首次加载可由 `simple-list` 或 `current-currency` 接口补齐。

## 4.3 统一金额显示封装
新增通用工具（建议）：
- `formatStoreMoney(amountInFen, options?)`
- 入参包含：金额（分）、门店货币、保留位数
- 输出示例：`$12.50`、`៛50,000`

替换范围优先级：
1. 商城统计中心（会员/商品/交易）
2. 订单中心、售后、分销
3. 会员中心金额类页面
4. 营销中心金额相关页面

> 说明：你当前系统已大量使用 `fenToYuan` + 固定 `¥` 文案，本次建议逐步替换为“门店货币格式化函数 + 动态符号”。

## 4.4 兼容策略
- 若门店无货币字段（空值），前端展示回退 `USD/$`。
- 不影响现有查询条件和 `storeId` 联动。

## 五、移动端改造（lukdo-mall）

## 5.1 门店货币获取
基于当前“租户 + 门店识别”链路，新增货币读取：
- 方案A（推荐）：门店识别后调用 `current-currency` 接口缓存。
- 方案B：复用门店信息接口返回货币字段。

缓存位置：
- `app store`（如 `app.currencyCode/app.currencySymbol`）
- 本地持久化（可选，切门店时刷新）

## 5.2 统一金额显示
封装 `formatStoreMoney`（JS 工具函数），所有价格显示统一走该函数：
- 商品价格
- 购物车
- 订单列表/详情
- 优惠券/营销活动金额
- 钱包与充值相关页面

## 5.3 切门店联动
- 门店变化时同步刷新货币缓存。
- 页面金额组件使用响应式货币状态，避免旧币种残留。

## 六、实施步骤（建议）

1. 数据库阶段
- 新增 `17-store-currency.sql` + rollback
- 执行回填

2. 后端阶段
- `StoreDO/VO/API` 扩展字段
- 保存校验与默认值
- 新增 `current-currency` 接口（可选但推荐）

3. 管理端阶段
- 门店管理页增加货币配置
- `StorePanel` 选中门店时写入全局货币
- 统计中心先完成动态货币展示替换

4. 移动端阶段
- 门店识别后读取货币
- 金额组件统一替换为动态货币格式化

5. 联调与验收
- 单门店、跨门店切换、无货币回退场景

- [ ] 门店管理可配置货币代码/符号并保存。
- [ ] 接口返回门店货币字段（后台与移动端都可获取）。
- [ ] 管理端切换门店后，统计中心金额符号同步变化。
- [ ] 移动端进入不同门店，商品与订单金额符号正确变化。
- [ ] 无货币配置时回退默认值（`USD/$`）。
- [ ] 不影响原有 `storeId` 隔离与查询逻辑。

## 八、风险与注意事项

1. 历史页面硬编码 `¥`
- 风险：改造不全会出现同页多种货币符号。
- 处理：按模块分批替换并增加扫描检查。

2. `fenToYuan` 直接拼接文案
- 风险：金额值正确但币种错误。
- 处理：封装统一格式化函数，禁止新代码直接拼接固定符号。

3. 门店切换后缓存未更新
- 风险：显示延迟或残留旧货币。
- 处理：切门店时强制刷新全局货币状态。

4. 多语言场景
- 风险：货币符号与本地化格式冲突。
- 处理：当前先做符号级展示，后续可扩展 `Intl.NumberFormat(locale, { currency })`。
